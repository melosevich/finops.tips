## What It Is

`HeadObject` is an S3 API call that retrieves metadata about an object (size, content-type, ETag, last-modified) without downloading its contents. It's in the same pricing tier as `GetObject`: **$0.0004 per 1,000 requests** in us-east-1. In CUR, it appears under `lineItem/ProductCode` = `AmazonS3` with `lineItem/Operation` = `HeadObject`. In FOCUS, look for `serviceName` = `Amazon Simple Storage Service` and `operation` = `HeadObject`.

Related metadata calls like `ListObjects`, `ListObjectsV2`, and `GetBucketAcl` fall into the same request tier. `ListObjects` operations are actually slightly cheaper: **$0.005 per 1,000 LIST requests** â€” but that's still 5Ă— more expensive per call than GET/HEAD on a per-request basis.

## Why It Matters

`HeadObject` calls are often generated by framework internals and infrastructure tools rather than application logic â€” making them hard to spot. Common surprises:

- **Existence checks before every write** â€” applications that call `HeadObject` to check whether a file exists before uploading it double the S3 request count for every write path. At 50M writes/day, that's 100M additional requests = **$40/day** in preventable charges.
- **Static site generators and deployment tools** â€” tools like `aws s3 sync` call `HeadObject` on every file to compare ETags before deciding whether to upload. On large deployments (10,000+ files), this can generate tens of thousands of metadata requests per deploy.
- **ML training pipelines** checking dataset completeness â€” a pipeline that iterates through 100,000 training files and calls `HeadObject` on each to verify presence before training generates 100K metadata calls on every run.
- **Health check or monitoring scripts** that poll S3 object existence or metadata on a schedule.
- **Terraform and IaC tools** â€” each plan/apply cycle reads S3 state backends and may call `HeadObject` to check for lock files, state files, and output files.

## How to Act

1. **Query CUR/FOCUS** for S3 request costs grouped by bucket and operation type. High `HeadObject` counts on buckets that shouldn't have high read traffic indicate a framework or tooling issue.
2. **Enable S3 request metrics** (per-bucket CloudWatch metrics) to get a breakdown of operation types. Standard S3 metrics in the console only show total requests.
3. **Eliminate redundant existence checks** â€” in application code, use conditional writes (`PutObject` with `If-None-Match: *`) or catch `NoSuchKey` errors on `GetObject` rather than calling `HeadObject` first.
4. **Cache metadata results** â€” if your application needs to repeatedly check the same object's metadata, cache the `HeadObject` result with a TTL appropriate to how often the object changes.
5. **Tune `aws s3 sync`** â€” for deployment pipelines, consider using `--exact-timestamps` or `--size-only` to reduce comparison operations, or switch to a manifest-based deployment approach that skips unchanged files without per-file HEAD checks.
6. **For LIST-heavy workloads**, structure your S3 key prefix hierarchy to enable efficient listing of specific subsets rather than listing large prefixes and filtering client-side.

## Example

A data platform runs hourly ETL jobs that use `aws s3 sync` to check 50,000 files for updates before processing. Each run generates ~50,000 `HeadObject` calls. Across 24 runs/day: 1.2M calls/day = 36M/month = **$14.40/month** just for sync metadata checks. Switching to a manifest file approach (one `GetObject` per run to fetch a list of changed files, then targeted downloads) reduces the monthly metadata cost to under **$1/month**.
